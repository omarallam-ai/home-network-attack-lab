# WiFi Cracking — Methodology & Password Strength Analysis

**Attack:** WPA2 Handshake Offline Cracking  
**Outcome:** Wordlist exhausted — password not found  
**Raw handshake file:** Excluded from repository  

---

## What This Documents

This documents the cracking methodology I applied, why the wordlist attack failed,
and a structured analysis of what a real attacker would do next — including estimated
GPU cracking times. The goal is to understand not just whether a password cracked,
but *why* it did or didn't, and what that means defensively.

---

## Phase 1 — Wordlist Attack (aircrack-ng)

### Wordlist used

The attack used a standard wordlist — the kind an attacker would reach for first:
common passwords, dictionary words, names, dates, keyboard walks (qwerty123, etc.),
and known leaked credentials from public breach databases.

### Why it failed

A wordlist attack is only as good as the words in it. It will find passwords that:
- Are common or previously leaked
- Are dictionary words with simple substitutions (p@ssw0rd, s3cur1ty)
- Follow predictable patterns (name + year, word + 123)

It will not find passwords that are long, random, or use uncommon character combinations
with no dictionary basis. The password used on this network fell into that category —
the wordlist exhausted without a match, which is itself useful information: the password
has no obvious dictionary relationship.

## Phase 2 — What a Real Attacker Would Do Next

### Rule-based attacks

Before brute force, a real attacker applies mangling rules to the wordlist — transforming
each word by capitalising, appending digits, substituting characters, reversing, and
combining words. Tools like hashcat support thousands of rules simultaneously.

This still requires the base password to have *some* dictionary origin to succeed.
Against a truly random password, rules provide no advantage.

### Mask attack (targeted brute force)

If rules fail, a hashcat mask attack systematically tries every combination within a
defined character set and length. The attacker makes educated guesses about the password
structure to narrow the keyspace.

Example masks and estimated cracking speed on a mid-range consumer GPU (RTX 3060,
~200,000 WPA2 candidates/second):

| Mask | Example | Keyspace | Estimated time |
|------|---------|----------|----------------|
| 8 lowercase letters | `abcdefgh` | 208 billion | ~12 days |
| 8 mixed case + digits | `Password1` | 218 trillion | ~35 years |
| 10 mixed case + digits | `Password123` | 3.7 quadrillion | far exceeds practical |
| 12 random mixed + symbols | `xK#9mL@2qP!v` | effectively infinite | not feasible |

**Key finding:** WPA2 uses PBKDF2-SHA1 with 4,096 iterations and the SSID as salt.
This intentional slowness (compared to raw MD5/SHA1) is why even a high-end GPU
manages only ~200K–1M guesses/second rather than billions. It's the main reason
a strong WPA2 password is practically uncrackable by brute force.

### Rainbow tables

Precomputed rainbow tables for WPA2 do exist for common SSIDs, but the SSID is part
of the PBKDF2 salt — so tables computed for "Linksys" are useless against "HomeNetwork42".
Non-default SSIDs effectively defeat rainbow table attacks.

## Defensive Conclusion

The outcome of this experiment confirms the threat model:

- A password with any dictionary basis will likely fall to a wordlist or rule attack
  within hours on commodity hardware
- A short password (under ~10 characters), even if random, is feasible to brute force
  with a GPU and enough time
- A long (16+ character), truly random password is not crackable in any practical timeframe
  against WPA2's key derivation function
- WPA3/SAE removes offline cracking entirely — the handshake equivalent cannot be attacked
  offline regardless of password strength

**Recommended minimum:** 20-character random passphrase generated by a password manager.
At that length, brute force against WPA2 is not a meaningful threat on any hardware
available today.

## Tools Reference

| Tool | Role | Command pattern |

| aircrack-ng | CPU wordlist cracking | `aircrack-ng -w wordlist.txt capture.cap` |
